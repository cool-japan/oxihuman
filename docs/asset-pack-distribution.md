## Overview

OxiHuman uses the **OXP** (OxiHuman Pack) format for distributing body morph presets,
PBR material definitions, and texture assets as self-contained, integrity-verified archives.

### OXP Archive Format

Each OXP distribution unit is a `.oxp.tar.zst` archive containing:

| Path inside archive | Contents |
|---|---|
| `manifest.json` | Pack metadata: name, version, author, license, SHA-256 digest tree |
| `presets/*.json` | Named body morph parameter sets (see `MorphPreset`) |
| `materials/*.json` | PBR material definitions (see `MaterialDef`) |
| `textures/*.{png,jpg,exr}` | Texture assets (see `TextureAsset`) |
| `targets/*.target` | Raw morph target deltas (binary) |
| `signature.json` | Ed25519 detached signature over the manifest hash (optional but recommended) |

The inner binary representation (prior to Zstandard compression) is produced by
`oxihuman_core::pack_distribute::PackBuilder`. Each pack carries an HMAC-SHA-256
integrity tag that is verified by `PackVerifier::verify_integrity` before any
deserialization occurs.

## Alpha Distribution Model

During the alpha lifecycle (v0.1.x), the primary distribution channel is **GitHub Releases**
on the `cool-japan/oxihuman` repository.

### Release Artifacts

Every tagged release produces:

```
oxihuman-alpha-pack-<version>.oxp.tar.zst
oxihuman-alpha-pack-<version>.oxp.tar.zst.sha256
oxihuman-alpha-pack-<version>.dist-manifest.json
```

The `.sha256` file contains the hex-encoded SHA-256 hash of the archive, suitable for
direct verification with `sha256sum -c`.

The `.dist-manifest.json` file is generated by
`oxihuman_core::asset_pack_builder::generate_distribution_manifest` and lists the
SHA-256 hash of every individual file inside the extracted pack directory. This enables
incremental synchronisation and fine-grained integrity checking without re-downloading
the full archive.

### How to Obtain a Pack

```bash
# Download via GitHub CLI
gh release download v0.1.1 \
  --repo cool-japan/oxihuman \
  --pattern '*.oxp.tar.zst*'

# Verify the archive checksum
sha256sum -c oxihuman-alpha-pack-0.1.1.oxp.tar.zst.sha256

# Extract
tar --zstd -xf oxihuman-alpha-pack-0.1.1.oxp.tar.zst

# Verify all individual files against the dist manifest
oxihuman pack-verify-dist \
  --manifest oxihuman-alpha-pack-0.1.1.dist-manifest.json \
  --pack-dir ./oxihuman-alpha-pack-0.1.1/
```

## Pack Signing Workflow

Packs should be signed before publishing so end users can verify provenance
independently of the transport layer.

### Signing a Pack Directory

```bash
# Generate a distribution manifest (SHA-256 per-file index)
oxihuman pack-dist-manifest --pack-dir ./my-pack/ > my-pack.dist-manifest.json

# Sign the pack directory (produces an Ed25519 signature over the manifest hash)
oxihuman sign-pack \
  --pack-dir ./my-pack/ \
  --key "$OXIHUMAN_SIGNING_KEY" \
  --signer-id "cooljapan-release-bot" \
  --output my-pack.signature.json
```

### Verifying a Signed Pack

```bash
# Verify the Ed25519 signature
oxihuman verify-sign \
  --pack-dir ./my-pack/ \
  --sig-file my-pack.signature.json \
  --key "$OXIHUMAN_PUBLIC_KEY"

# Additionally verify file-level hashes
oxihuman pack-verify-dist \
  --manifest my-pack.dist-manifest.json \
  --pack-dir ./my-pack/
```

The signing implementation lives in `oxihuman_core::pack_sign`. The CLI subcommands
`sign-pack` and `verify-sign` delegate to `sign_pack_dir` and `verify_pack_signature`
respectively.

## CDN Strategy

### Primary: GitHub Releases

GitHub Releases act as the authoritative, immutable distribution source for all
officially tagged packs. Every asset attached to a release is content-addressed by
its SHA-256 hash in the accompanying `.dist-manifest.json`.

Advantages:
- Globally cached via GitHub's CDN (Fastly).
- Immutable — release assets cannot be silently replaced.
- Free for open-source projects.

### Secondary: IPFS Content Addressing

For long-term archival and decentralised access, each `.oxp.tar.zst` archive is pinned
on IPFS after release. The CID (Content Identifier) is derived from the archive's raw
bytes and is included in the dist-manifest under the `ipfs_cid` field (v0.1.1+).

```json
{
  "schema_version": "0.1.1",
  "ipfs_cid": "bafybeig...",
  "files": { ... }
}
```

Users with IPFS access can retrieve packs by CID:

```bash
ipfs get bafybeig... -o oxihuman-alpha-pack.oxp.tar.zst
```

## Version Compatibility

### OXP Manifest Schema Version

The `schema_version` field in the dist-manifest JSON tracks breaking changes to the
manifest structure:

| `schema_version` | OxiHuman release | Notes |
|---|---|---|
| `"0.1.0"` | v0.1.0 | Initial alpha; no IPFS CID field |
| `"0.1.1"` | v0.1.1 | Added `ipfs_cid`; per-file dist manifest functions |

### Schema Migration

The `oxihuman_core::version_migration` module provides `migrate_schema` for upgrading
persisted session and manifest files between schema versions. Pack consumers should
call `migrate_schema` when loading manifests from older releases.

## Privacy Model

OxiHuman's distribution tooling is designed to be **offline-first** with no telemetry
and no centralised authentication requirements.

### Client-Side Hash Verification

All integrity checks use SHA-256 computed locally from the raw file bytes. No hash is
ever transmitted to a remote server during the verification step.

### No Telemetry

OxiHuman CLI does not phone home, log usage metrics, or embed tracking identifiers in
generated files or download requests.

### Offline Operation

`pack-dist-manifest` and `pack-verify-dist` operate entirely on local files. An internet
connection is only required for the initial download from GitHub Releases or IPFS.

## Usage Examples

### Generate a Distribution Manifest

```bash
# Print the manifest JSON to stdout
oxihuman pack-dist-manifest --pack-dir ./my-pack/

# Save to file
oxihuman pack-dist-manifest --pack-dir ./my-pack/ > my-pack.dist-manifest.json
```

The output is a JSON object with a `schema_version` field and a `files` object that maps
relative file paths to their hex-encoded SHA-256 hashes:

```json
{
  "schema_version": "0.1.1",
  "files": {
    "manifest.json": "a3f5...",
    "presets/athletic.json": "7c1d...",
    "materials/skin.json": "b8e2..."
  }
}
```

### Verify a Distribution Manifest

```bash
oxihuman pack-verify-dist \
  --manifest ./my-pack.dist-manifest.json \
  --pack-dir ./my-pack/
# → Manifest verification: OK
```

If any file has been modified or is missing, the command prints `Manifest verification: FAILED`
to stderr and exits with code 1.

### Full Round-Trip Example

```bash
# 1. Build a pack from source assets
oxihuman pack-build \
  --targets ./targets/ \
  --output ./my-pack/manifest.toml

# 2. Generate the distribution manifest
oxihuman pack-dist-manifest --pack-dir ./my-pack/ > ./my-pack.dist-manifest.json

# 3. Sign the pack
oxihuman sign-pack \
  --pack-dir ./my-pack/ \
  --key "$SIGNING_KEY" \
  --signer-id "release-bot" \
  --output ./my-pack.signature.json

# 4. Upload to GitHub Release (with gh CLI)
gh release upload v0.1.1 \
  my-pack.oxp.tar.zst \
  my-pack.dist-manifest.json \
  my-pack.signature.json

# 5. End-user verification
oxihuman pack-verify-dist \
  --manifest my-pack.dist-manifest.json \
  --pack-dir ./my-pack/

oxihuman verify-sign \
  --pack-dir ./my-pack/ \
  --sig-file my-pack.signature.json \
  --key "$PUBLIC_KEY"
```
